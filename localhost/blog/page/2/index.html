Installing Sumatra 0.6-dev in Ubuntu 13.10

Sumatra is a great tool for reproducibility and provenance when you are running numerical simulations. Keeping your work reproducible ensures that you and others will be able to check your results at a later time, while provenance is to keep track of where your results came from in the first place.

I won’t get into details about using Sumatra in this post, as its documentation is quite good at describing that already.

Computational Physics members

If you are a member of the Computational Physics group and have access to the computing cluster, Smaug, you don’t have to install anything. Just ssh into our dedicated Sumatra machine, named bender, and run your jobs from there. In the future, all machines will support Sumatra, but for now, we only have one dedicated machine for this task:

ssh bender

On your own machine

Otherwise, or if you want it on your own machine, you will have to install it manually. This is done by cloning the repository and running the setup.py file. This will install Sumatra with all dependencies:

hg clone https://bitbucket.org/apdavison/sumatra
cd sumatra
python setup.py install --user

Adding –user to the final command ensures that all packages are installed to your home directory, under .local/lib, so that  you don’t have to install with sudo privileges, and it makes it a bit easier to remove the package if you don’t want to keep it.

We also need the GitPython and PyYAML modules, which you may install using apt-get:

sudo apt-get install python-git python-yaml

And that’s it! You should now be able to run your projects with Sumatra.

Automatic panel lettering within LaTeX

Leave a reply

Frequently when you include figures in a document you want them as panels within one larger float. Standard procedure is to put a letter on each panel and refer to it like Figure 69g. If you have unlimited space, using the subcaption or subfig packages to assign a letter and caption to each panel is the way to go.

However, if space is limited you usually want a single letter inside the panel. Your options are 1) open the image/plot/whatever in image editing software and put a letter on it; 2) have your plotting routine put the letter on; 3) put the letter on in LaTeX. 1) is a bad idea if you expect the contents of the panel to change, because you have to redo the lettering every time. 2) works better, but gets annoying when you shuffle your panels around, or when you use the same panel in different figures (often in a different document altogether). Here I show you how to do 3).

The idea is simple: use subfig to automatically assign a letter to your panel, but don’t ask for a caption. Refer to the panel number assigned, but do this within a \put() command, and put the letter returned by \subref somewhere on the panel (I opted for top left, but you can easily change this or move the letter around). You’re done.

The figure looks like you expect, and all the subfig functionality is unaffected.


\begin{figure}
\panel{.5}{./panel1}{pan:cod}
\panel{.5}{./panel2}{pan:gull}
\caption{Two panels from separate image files with the subfig package and some other commands used to place panel lettering on the panels within LaTeX. \protect\subref{pan:cod} A cod. \protect\subref{pan:gull} A gull.\label{fig:animals}}
\end{figure}

Working example for download.

Note that contrary to what Filip said in a recent post, subfig is still going strong.

Structuring large(r) projects using QMake

Leave a reply

Note: This post overlaps / conflicts with some of the material dicussed here. So I excourage you to read both posts if you have not read the previous one already.

Hey champs,

first of all, let me just say that if you’re not using Qt Creator for organizing and coding your projects, which uses QMake intrinsically, the level of this post will be somewhat higher. Nevertheless, I’ll try to keep to the basics and explain things in a simply way, without all the jazz. If you are interested in the jazz, however, I am very interested in discussing it in a separate post or in the comments / over some coffee.

So let’s get to it! This post is aimed at those who develop a code centered around the use solvers, i.e. one implementation of a specific algorithm. Examples could be Molecular Dynamics (MD), Variational Monte-Carlo, Coupled-Cluster, Hartree-Fock, and so on.

Structure your code. Don’t mash all your files in the same folder. It’s ugly. I don’t care if you are using an IDE which displays them in an organized fashion – it’s still ugly.

Separate the source code from the rest of the project. Let’s say you’re coding MD. You may use your implementation to study numerous systems, e.g. basic tempering of Argon, phase coexistence simulations, and so on. A brute force way of implementing multiple uses of a solver is to switch test in the main file and finally simply call run() using whichever objects was initialized in the switch.

However, I will argue that this is a bad way of doing things. Your main file “main.cpp” will be awfully cluttered and a hazzle to work with. I would rather suggest you create a separate main file for each of the separate uses of the solver, e.g.

  • …/MD/apps/argon/argonmain.cpp
  • …/MD/apps/phasecoexistence/phasecoexistencemain.cpp

and put your source code in

  • …/MD/src/

Moreover, if your classes have subclass implementations, they shold be structured in the following way for maximum transparency:

  • …/MD/src/superclass/superclass.h
  • …/MD/src/superclass/superclass.cpp
  • …/MD/src/superclass/subclass/subclass.h
  • …/MD/src/superclass/subclass/subclass.cpp

Multiple main files require multiple executables, as each executable is linked to one specific main function. However, each of the main function reuse the same code for the algorithm. It is thus recommended to compile the source code to a shared library. You can think if a library as an implementation with the main file removed; exactly what we need.

Another argument for using shared libraries, is that if you do unit testing, you are guaranteed that the same machine code is used for testing and production runs.

Your library file should be placed in

  • …/MD/lib/

The last missing piece now is how to include the library headers in the main files. The brute force way of doing this is to implement every single header that you need. This is unnecessary. Since the library is built already, there is no advantage to gain by not including every single header. So you should make a combined header

  • …/MD/include/MD.h

which includes everything you need to use the library to it’s fullest. For example:


#pragma once

#include “../src/stuff.h”

#include “../src/superclass/subclass/subclass.h”

…


such that your main files can start like the following:


#include <MD.h>

int main() {

…

}


Do note that this header should never be used within the library itself, only by outside sources which link to the library directly (apps).

So to recap, we’ve got some layers of structuring going on now:

  • …/MD/src/                    – source code going into the MD solver
  • …/MD/lib/                     – compiled MD shared library
  • …/MD/include/MD.h    – complete header for use in main.
  • …/MD/apps/                 – specific implementations.

Setting up QMake for this structure is very simple. First of all we need

  • …/MD/MD.pro

TEMPLATE = subdirs
CONFIG += ordered

SUBDIRS += src apps

OTHER_FILES +=  $$PWD/include/MD.h


This simply says that we are bulding multiple things; one thing located in src, and another thing located in apps. We are also keeping track of the file MD.h, but we are not using it to build anything.

then we need

  • …/MD/src/src.pro

include(../defaults.pri)

TEMPLATE = lib
TARGET = ../lib/MD

HEADERS = stuff.h superclass/superclass.h …

SOURCES = superclass/superclass.cpp …


where the defaults.pri file includes all the compiler information etc. that is general for everything you build. An example of this is listed here.

finally, we need

  • …/MD/apps/apps.pro

TEMPLATE = subdirs
CONFIG += ordered

SUBDIRS += argon phasecoexistence …


Implementing an app is then simply a matter of adding another subdir to the list of apps.pro, and setting up e.g.

  • …/MD/apps/argon/argon.pro

include(../app_defaults.pri)

SOURCES = argonmain.cpp


where

  • …/MD/apps/app_defaults.pri

looks something like this:

include(../defaults.pri)

TEMPLATE = app
CONFIG += console

INCLUDEPATH += $$TOP_PWD/include

LIBS += -L$$TOP_PWD/lib -lMD


where TOP_PWD is set in defaults.pri and references the top MD folder. All in all, this project file setup should enable the easy use of the library as introduced previously.

Using this structure encourages external use of your code since it is already set up in such a way that it does not require recompilation and a messy ocean of headers to use. To ensure that all added apps can function independent of the source (and thus more easily usable by others) the following lines can be added to the source project file, which ensures that your compiled library file will not remain at a distant build directory which other apps may not know about:

!equals(PWD, $${OUT_PWD}) {
QMAKE_POST_LINK += $(COPY_DIR) $$OUT_PWD/../lib $$TOP_PWD
}

Moreover, you should enable debug symbols for your library, so that the debugger will be able to speficy where in your code it crashed (also in src.pro):

QMAKE_LFLAGS += -g

this is not default. however, you may keep this only for debug mode.

For examples of setups using the methods introduced above, see for example my quantum Monte-Carlo code, or my kinetic Monte-Carlo code.

Including vector graphics in LaTeX using includesvg

Leave a reply

While writing my master thesis I’ve found that one of the best ways of explaining a lot of concepts is by using illustrations and figures. I make most of my illustrations using Inkscape, an open source program for making and editing vector graphics.

Using vector graphics mean that all lines in the image is stored as mathematical expressions, contrary to raster graphics where an image is stored by storing the color in each pixel. This means that if you try to zoom in on a raster image, you will get and ugly pixelated image, while if zooming in on a vector image the computer can produce a higher resolution image from the mathematical expressions, as seen in the figure below.

This image illustrates the difference between bitmap and vector images. The bitmap image is composed of a fixed set of dots, while the vector image is composed of a fixed set of shapes. In the picture, scaling the bitmap reveals the dots while scaling the vector image preserves the shapes. Source.

The other advantage of vector images is that if the image consists of a few simple objects, as is the case for most illustrations in my thesis, the image can be stored in a very compact way. This means that the resulting pdf will be smaller in size, while still having nice and high resolution figures.

Including vector graphics in LaTeX

The impatient can check out a working example in this Github repo.

The problem is that there hasn’t been any really good ways of including vector graphics in LaTeX documents. But I recently discovered the package svg, which greatly simplifies everything. Using this package you can easily include vector graphics stored in the svg-format, which is the default format in Inkscape, by simply using the command

\includesvg{example} % Note that the file ending is omitted!

if the name of your image is “example.svg”, and adding

\usepackage{svg}

to your preamble. Note that this only works if “example.svg” is in the same directory as your LaTeX source file (I will come back to how to include images in other locations).

To compile the document you have to allow calling of external functions during compilation, which you do with latex or pdflatex by adding the flag “–shell-escape”:

pdflatex --shell-escape main.tex

The way this works is that it uses Inkscape to export the file to a .pdf_tex and a .pdf file. The advantage of doing this is that all text in the figure will have the same formatting as the rest of your document, and that the font can be controlled directly in your LaTeX source file.

One other advantage of using \includesvg{} is that you don’t have to export your figures from Inkscape every time you make a change to get the new image in your pdf, since the svg package does this automatically. The package also knows to only re-export an image if you changed the svg-file.

The package documentation does a good job of describing usage and options of the package. An often used option is the width or height of the image, which you control in the usual way

\includesvg[width = 0.7\textwidth]{example}

Installing the svg package

I don’t think the svg package is available in any Ubuntu-package yet, but installing it is as simple as downloading svg.sty and putting it in your texmf path

$HOME/texmf/tex/latex/svg

or putting it in the same directory as your LaTeX source file. You also have to have Inkscape installed, and possibly also imagemagick.

Inkscape export options

The default export option the package uses is to export the page of your svg. What you usually want is to export the drawing, so we change this by adding the command

\setsvg{inkscape = inkscape -z -D}

in the preamble.

Including files in other locations

Most of you probably have your images in a different location than your LaTeX source code. The way Inkscape exports svg-files to .pdf_tex and .pdf means that you have to use some special commands to include svg-files in other locations, but this is done by adding some options to the \includesvg{}-command. If your images are located in a directory called “images” in the same directory as your main LaTeX file you use

\includesvg[svgpath = images/]{example} % Note the "/" at the end

This path can also be set permanently using

\setsvg{svgpath = images/}

Subfig vs subcaption

(Take the section with below with a grain of salt, as according to a recent blog post here on this blog, the subfig package isn’t depreceated after all.)

The svg package uses the package subfig, which apparently is deprecated[1][2], and should be replaced by for example subcaption. Using subfig and subcaption in the same document doesn’t work very well, so if you don’t want to use subfig this can be solved by editing svg.sty. Replace “subfig” with “subcaption” on line 40 and you’re done. An working version that uses subcaption can be found here.

Simple cross-referencing in LaTeX using cleveref

Leave a reply

I recently discovered a LaTeX-package called cleveref, which simplifies cross-referencing when writing LaTeX-documents. Instead of writing

see fig. \ref{referencetoawesomething}

you can write

see \cref{referencetoawesomething}

cleveref will then figure out if what you’re referencing is a figure, equation, section etc., and translate the \cref{} command to the correct text, for example “fig. 2.5b”. You can also reference several things at once using

\cref{ref1,ref2}

which will be replaced with for example “figs. 2.5b and 2.5c”.

The output of the \cref{} command can naturally be tweaked to output “Figure 2.5b”, “Fig. 2.5b”, or anything else. cleveref also has support for both English and Norwegian, among others.

To use the package you install it in Ubuntu/Linux Mint with

sudo apt-get install texlive-latex-extra

and add the following to your preamble

\usepackage{cleveref}

See the cleveref package documentation for more information and options.

New project structure for projects in Qt Creator with unit tests

Note: This is a new version of an earlier post, with a revised project structure.

Note 2: See this post for the same project structure using the even better Catch testing framework.

This post assumes that you are using a C++ testing framework such as UnitTest++. See this earlier post for some information on installing UnitTest++ on Ubuntu.

To get the most out UnitTest++ it is a good idea to integrate its output into the Qt Creator IDE. The way I have set this up in Qt Creator is with subprojects. One for the main project, which again is split into the app itself and a library, and one for the tests. In addition, I have a helper project file, named defaults.pri. The structure of the project is like this:

MyProject
├─ MyProject.pro
├─ defaults.pri
├─ app/
│  ├─ app.pro
│  └─ main.cpp
├─ src/
│  ├─ src.pro
│  └─ myclass.cpp
└─ tests/
   ├─ tests.pro
   └─ main.cpp

An example project using this code structure has been posted on Github by Filip Sund. (Thanks to Filip for doing this!)

The main project file, MyProject.pro will now be based on a subdirs template, and may look like this:

TEMPLATE = subdirs
CONFIG+=ordered
SUBDIRS = \
    src \
    app \
    tests
app.depends = src
tests.depends = src

The app.depends and tests.depends statements makes sure that the src project is compiled before the application and tests, because the src directory contains the library that will be used by both the app and the tests.

(Thanks to Will for noting that this works better for parallel builds with make -j 8 than my previous version only using CONFIG+=ordered. We should keep CONFIG+=ordered in there still, though, because depends doesn’t affect the order when using make install).

defaults.pri

Each of the other .pro files will include defaults.pri to have all the headers available. defaults.pri contains the following:

INCLUDEPATH += $$PWD/src
SRC_DIR = $$PWD

If the library, main program and tests use common libraries, it is very useful to have the defaults.pri define these dependencies too.

./src

In the src folder, I have myclass.cpp, which is the class that I want to use and test. The src.pro needs to compile to a library, so that it may be used both by app and tests, and could look something like this:

include(../defaults.pri)
CONFIG -= qt

TARGET = myapp
TEMPLATE = lib

SOURCES += myclass.cpp
HEADERS += myclass.h

What this class does is not so interesting, it could be anything. A simple example could be this header file:

#ifndef MYCLASS_H
#define MYCLASS_H

class MyClass {
public:
    double addition(double a, double b);
};

#endif // MYCLASS_H

With this accompaning source file:

#include "myclass.h"

double MyClass::addition(double a, double b) {
    return a * b;
}

./app

I only have a main.cpp file in app, because the app is basically just something that uses everything in the src folder. It will depend on the shared compiled library from src, and app.pro would look something like this:

include(../defaults.pri)

TEMPLATE = app
CONFIG += console
CONFIG -= app_bundle
CONFIG -= qt

SOURCES += main.cpp

LIBS += -L../src -lmyapp

The main.cpp file could be a simple program that uses MyClass:

#include <myclass.h>
#include <iostream>

using namespace std;

int main()
{
    MyClass adder;
    cout << adder.addition(10, 20) << endl;
    return 0;
}

./tests

In the tests folder I have simply added a main.cpp which will run the tests. Then tests.pro has the following contents:

include(../defaults.pri)
TEMPLATE = app

CONFIG   += console
CONFIG   -= app_bundle
CONFIG   -= qt

SOURCES += main.cpp

LIBS += -lunittest++ -L../src -lmyapp

Which now links to the myapp library in addition to the unit tests.

The main.cpp in tests file which could contain the following, if we were to use UnitTest++ as our testing library:

#include <unittest++/UnitTest++.h>
#include <myclass.h>

TEST(MyMath) {
    MyClass my;
    CHECK(my.addition(3,4) == 7);
}

int main()
{
    return UnitTest::RunAllTests();
}

This test will fail because my implementation of MyClass::addition is completely wrong:

class MyClass {
public:
    double addition(double a, double b) {
        return a * b;
    }
};

Note that I’m including MyClass by through <myclass.h> which is possible because of the INCLUDEPATH variable in defaults.pri.

This is hopefully all you’ll need to define a project that compiles a library, as well as tests and an application using the library.

Simplistic Config Files for C++

Leave a reply

Hey all,

as many of you know, as codes tend to increase in size, they also (hopefully) tend to become more versatile and more reliant on various parameters to perform a specific task. At this limit, it becomes very dull having to open the code, recompile the code, and all that jazz just to test a new case. In light of this, a very common way to handle all sorts of parameters is through so-called configuration files, or config files for short.

There exists many standards when it comes to config files, however, common to all is that they work around cell blocks. A typical setup would be something like this:

FirstCell =
{
    CellInFirstCell =
    {
        someVariable = 2;
    }

}

SecondCell = ...

Using interpreted languages like Python, one can use a config library (there exist plenty) which stores all the data in an object from which we can access all the data in the following manner:

retrievedData = FirstCell.CellInFirstCell.someVariable

Creating this library could be a simple project in lower level programming courses.

For compiled languages, on the other hand, it is not as easy, as the content of the file is not known compile time. Hence we are more reliant on a rigorous yet simple config file library.

The most commonly known cross-language library is json, however, personally, I find that if you do not need the super-duper cross-language support, libconfig++ serves as a more clean and elegant library (this is of course open to discussion).

Libconfig for c++ is available through apt-get and is a part of the comp-phys package. If you are not using this package, you can get it by typing

sudo apt-get install libconfig++-dev

Parsing the first config file example (which is in libconfig syntax) can be done in the following manner in your c++ program:

#include 

using namespace libconfig;

int main()
{

    Config cfg;

    cfg.readFile("name_of_config_file");

    const Setting & root = cfg.getRoot();

    int retrievedData = root["FirstCell"]["CellInFirstCell"]["someVariable"];

    ...
    return 0;
}

For transparency you could also load entire cells:

const Setting & FirstCell = root["FirstCell"];
int retrievedData = FirstCell["CellInFirstCell"]["someVariable"];

This is good and all, but what if you have a typo, or type mismatch (e.g. try loading a double into an int)? This brings me to the second topic of this post: Exception handling with libconfig.
If libconfig fails to load a value for some reason, it will simply throw an exception explaining the general nature of the error, that is, either setting not found or type mismatch. The tedious part is that it says nothing about which value failed, neither is it very easy to tell the exceptions apart.

After using the library for some months I decided to do something about this, namely write a wrapper for libconfig which does the exception handling rigorously. You can find the full story on , but here’s the short version:

Take the first example use of libconfig. Now, instead of including libconfig directly, we include the utilities header and retrieve the value using the getSetting function:

#include 

using namespace libconfig;

int main()
{

    Config cfg;

    cfg.readFile("config.cfg");

    const Setting & root = cfg.getRoot();

    int someVariable = getSetting(root, {"FirstCell",
                                         "CellInFirstCell",
                                         "someVariable"});

    ...
    return 0;
}

Note: The brackets requires compilation with c++ 11

At this point, there is nothing new going on. However, let’s introduce an error, replacing “someVariable” by “anotherVariable”. The following error output is then given runtime:

Unable to load config key 'anotherVariable' from
 root-->FirstCell-->CellInFirstCell-->anotherVariable
what() : SettingNotFoundException
 Mismatch in given setting name.

What is not covered in the previous example is what happens if the typecast fails, i.e. that libconfig finds a mismatch in the format of the variable in the file and the type requested; it simply crashes.

If you want to avoid this, a template type can be specified as follows:

int someVariable = getSetting<int>({"FirstCell",
                                   "CellInFirstCell",
                                   "someVariable"});

here in the case of a mismatch (i.e. if we set someVariable = 2.0) , the following message appears:

Unable to load config key 'someVariable' from
 root-->FirstCell-->CellInFirstCell-->someVariable
what() : SettingTypeException
 Mismatch in given template type and config file variable format.

Errors do not necessarily have to be at the bottom level, in which the lower levels will not be indicated:

Unable to load config key 'CellInFirstCell_with_typo' from
 root-->FirstCell-->CellInFirstCell_with_typo
what() : SettingNotFoundException
 Mismatch in given setting name.

Moreover, we often only want to retrieve a variable from one level down, which means that calling the getSetting function with a list is tedious. If you wish only to read a single variable from the next level, then you will not have to use lists, but can rather use the function getSurfaceSetting as follows:

...
    const Setting & root = cfg.getRoot();
    const Setting & mid  = getSetting(root, {"FirstCell", "CellInFirstCell"});

    int someVariable = getSurfaceSetting<int>(mid, "someVariable");
...

This is equivalent to dropping the ‘Surface’ and adding the string in brackets, but thought of as a more convenient option for single depth parsing.

Note also that in the last example, we saved a variable mid which is a simple libconfig++ Setting object. There is absolutely no difference in the memory address of mid and root[“FirstCell”][“CellInFirstCell”] by construction.

Regardless of whether you use intermediate storage of cells, the error message will trace back to the root object.

Anyways, that’s all for now! I hope you guys find a use for it; I use do!

Until next time, enjoy! 🙂

Jørgen out
